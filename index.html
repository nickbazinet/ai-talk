Why the magic isn't in asking the AI —

Nick Bazinet  ·  Sovra / F-35
Where We're Starting

It feels like everyone codes now.

They don't.

There's a huge gap between asking AI to build something and guiding it the right way. The first feels magical for ten minutes. The second is what actually scales.

The Distinction That Matters

Two ways to work with AI.

Approach A

Vibe coding

  • "Hey AI, build me a thing."
  • One-shot prompts, no rules
  • Output reflects whatever the model picks
  • Looks great in a demo
  • Falls apart at team scale
VS

Approach B

AI-assisted development

  • Build the harness around the AI
  • Hooks, rules, skills, agents
  • Output reflects your patterns
  • Feels boring. Looks great in a PR.
  • Scales across a team
Why bother

No human writes code this fast.

You can be skeptical of AI. You can be allergic to AI. The speed gap is still real — and the speed is where the value lives, once you can trust the output.

10×
faster than typing — if the harness is right.
The Key Idea

Guiding the AI isn't telling it what to write.

It's building the environment so the AI follows your patterns without you having to say it every time.

Treat the AI like a top-of-field engineer. They know how to code. They don't know your standards, your guidelines, your domain. That's your job.
A note on tools

Pick your weapon.

The pattern works with any of these. The mechanics differ, the principles don't. For this talk, we're using Claude Code.

Cursor Claude Code OpenCode Windsurf Aider … etc

Claude Code gives us four primitives to work with: hooks, rules, skills, and agents.

The four primitives

Your harness, in four parts.

01 / HOOKS

Always run.

Scripts that fire before or after a Claude action. Linters, SAST, fast checks. Their job: shrink the feedback loop.

02 / RULES

Path-triggered.

Instructions Claude reads when it touches certain files. The AI doesn't choose — the path does. Your business knowledge lives here.

03 / SKILLS

Workflows it picks.

Reusable workflows or knowledge Claude calls when relevant. Heavy on context. Best when short and reused, not invented.

04 / AGENTS

Specialized helpers.

Sub-agents for narrower jobs (review, research, planning). Not the focus today, but they round out the picture.

01 — Hooks

Hooks are non-negotiable checks.

  • Run before or after a Claude action
  • Should be fast — you'll feel the latency
  • Output (logs, errors) is fed back to the AI
  • Best for things that must happen: linters, SAST, format checks

Real example

Run the right linter based on file extension. Claude finds out about an error now — not after it pushes and waits ten minutes for GitLab CI.

Why hooks matter

The goal is always the same: shorten the feedback loop.

WRITE
AI generates
→
HOOK
Lint & check
→
FIX
AI sees error
→
SHIP
PR is clean

Every minute the AI is wrong without knowing it is a minute it's heading further in the wrong direction.

02 — Rules

Rules are where the magic starts.

A rule is just an instruction tied to a file path. Touch a file in /payments/? The payments rule fires. Always.

Unlike skills, the AI doesn't get a vote. Whether it actually follows the rule… that's the part you tune.

# .claude/rules/payments.md applies-to: "src/payments/**" When working in this directory: - Never log full card numbers - All amounts are integer cents - Use PaymentResult<T> for return types - Tax calc lives in tax/engine.ts
The secret of good rules

Don't teach the AI to code. Teach it your business.

Don't bother

  • "Write clean code"
  • "Follow SOLID principles"
  • "Use meaningful names"

It already knows. You're just burning tokens.

Worth every token

  • "Tenants are isolated by org_id; never query without it"
  • "All money is integer cents"
  • "FeatureFlag reads go through flagsClient, never env vars"

How do we measure if a rule is working? Honestly — we don't yet. It's developer feedback and the count of comments left on merge requests. That's a real open problem.

03 — Skills

Skills are workflows the AI picks up.

Where rules are triggered by you (via paths), skills are picked by the AI when the model decides one is relevant.

Repeatable workflow

"How we add a new API endpoint" — steps, conventions, tests.

Just-in-time knowledge

Domain context the AI only needs in specific situations.

Planning helpers

"Before you implement, plan first" — with what to consider.

Hard-won lessons

Two things to know about skills.

1   Don't trust AI-written skills (yet)

Auto-generated skills are way too verbose. They pack in information the AI doesn't need and confuse the one trying to use it.

Use AI to draft, then ruthlessly cut. Straight to the point wins.

2   Reuse before you reinvent

People have spent hours tuning skill libraries. Frameworks like bmad and superpowers are excellent and easy to plug in.

Start from someone else's work. Customize from there.

Project context

CLAUDE.md — less is more.

  • Cap it at ~200 lines.
  • Project-specific context only
  • Explain the high-level architecture
  • Don't explain how to write code
  • Don't explain coding conventions — the AI will pick those up from your codebase

My first CLAUDE.md was a brick. Most of it was wasted tokens on things the AI already knew.

# CLAUDE.md (the good kind) ## Architecture - Frontend: Next.js + tRPC - API: NestJS, deployed via GitLab CI - DB: Postgres, multi-tenant via org_id ## Domain quirks - Pricing engine lives in /pricing - Auth runs through @platform/auth - Audit log is append-only
Code review

When you ship 10+ MRs a day, review breaks.

One or two code-owners cannot eyeball that volume manually. Tools like CodeRabbit and GitLab Duo help, but…

In a perfect world, the review tool would know every business rule and catch every logic bug. We are not in that world. So we adapt.

What's working for us

Layered review, with custom rules.

01.   Local "core review"

Frameworks like bmad / superpowers run a structured review locally before the PR opens.

02.   Tool pass

CodeRabbit / GitLab Duo for the second pass. Custom rules, not stock config.

03.   The golden rule

If you write a comment by hand, also update the tool. Never repeat yourself.

The shift

The job isn't writing code anymore.

It's writing the harness that lets the AI write the code — well, in your style, at your standard.

That takes discipline. It's also where we've seen the biggest jump in quality.

Hidden corollary

AI inherits your tech debt — and amplifies it.

The AI learns your patterns from your codebase. If your code is all over the place, its output will be all over the place — just faster.

Counter-intuitively, AI makes refactoring more important, not less.

The good news

You're shipping features faster. Use the slack to pay down debt. That investment compounds — every future AI-generated line gets better.

Last principle

Chase the 80%. Ignore the last 20%.

AI doesn't know — it picks the most statistically relevant answer. It will never be perfect.

Make the AI 80% better at what your team needs. That's already a massive win. The last 20% will eat your whole quarter.
